# Review 1: opencode-waitfor

Date: 2026-07-10
Branch: feature/opencode-waitfor

## Summary

The implementation is functionally complete and well-structured. All 37 tests pass (9 files), the module architecture matches the spec (┬¦5) exactly, and the npm packaging is correct (12 files, no test/ or node_modules/ leakage). Every spec requirement ┬¦1-┬¦8 has a corresponding implementation. The code is clean, follows the TDD plan faithfully, and has zero runtime dependencies.

However, **the `typecheck` CI step is broken** ŌĆö `tsc --noEmit` fails on all 9 test files because the tsconfig is missing `bun-types` type definitions (a deviation from the plan's tsconfig spec). This is the only Critical issue. There are also 3 Major gaps in test coverage (truncation, array status codes, interval <= 0), and 3 Minor issues (diagnostics, validation redundancy, test casts).

**Issues: 1 Critical, 3 Major, 3 Minor**

---

## Issues

### Critical
- [C1] **`tsc --noEmit` fails ŌĆö tsconfig missing `bun-types`** ŌĆö `tsconfig.json:7`

  The current tsconfig has `"types": ["node"]` but the plan's spec (┬¦Task 1, Step 2) specifies `"types": ["node", "bun-types"]`. With `"include": ["src", "test"]`, TypeScript cannot resolve `bun:test` imports in all 9 test files:
  ```
  test/command.test.ts(1,40): error TS2307: Cannot find module 'bun:test'
  ```
  This makes the `typecheck` npm script unusable and would fail CI.
  
  **Fix options:**
  - **(a) Add `bun-types`** to devDependencies (`bun add -d bun-types@1.3.14`) and add it to the `types` array. This matches the plan's intent.
  - **(b) Exclude test files** from tsconfig (`"include": ["src"]`) ŌĆö tests still pass via `bun test` which understands `bun:test` natively. This is simpler but means test types aren't checked.

### Major
- [M1] **No test for response body truncation** ŌĆö `test/http.test.ts`, `test/command.test.ts`

  Spec ┬¦4 and the plan mandate 2048-byte truncation of HTTP response bodies and command stdout/stderr. The current tests use tiny responses/commands and never verify that `bodyPreview`, `stdoutPreview`, or `stderrPreview` are actually capped at `MAX_BYTES`. If the truncation logic were removed, no test would fail.

- [M2] **No test for `expect.status` as an array** ŌĆö `test/http.test.ts`

  The spec allows `status` to be a `number | number[]` (e.g. `[200, 201]` for multiple acceptable codes), and the `statusOk()` helper handles this via `Array.isArray()`. However, only the single-number case (`{ status: 200 }`) is tested. The array case has zero coverage.

- [M3] **No explicit test for `interval <= 0`** ŌĆö `test/validate.test.ts`

  The spec requires `interval <= 0` to throw an error, and `validate()` checks this at line 22. But the test suite only covers `timeout <= 0` (explicitly) and `interval > timeout` (covers the "interval > timeout" path, not the "interval <= 0" path). A test for `interval: 0` or `interval: -1` is missing.

### Minor
- [m1] **HTTP json_match mismatch snapshot lacks diagnostic fields** ŌĆö `src/probes/http.ts:41`

  When a `json_match` entry fails, the snapshot only includes `{ status, bodyPreview }`. It does NOT include *which* dot-path mismatched or the expected-vs-actual values. This means the agent must re-probe manually to diagnose which field failed (the exact problem the tool was built to solve ŌĆö see spec ┬¦1 evidence). The `last` payload in a timeout scenario would say "timed out" without telling you what was wrong with the last probe.

- [m2] **Redundant validation between Zod schema and `validate()` function** ŌĆö `src/index.ts:26-29,53`, `src/validate.ts:19-23`

  The Zod schema marks `timeout` and `interval` as `.positive()`, so the opencode host would reject non-positive values before `execute()` runs. The `validate()` function re-checks `timeout <= 0` and `interval <= 0` with hand-crafted error messages. This isn't harmful but creates two sources of truth for input validation. If future changes only update one path, behavior could become inconsistent.

- [m3] **Heavy `as never` casts in integration test** ŌĆö `test/integration.test.ts:2,5,7,9,11,14`

  The integration test uses `as never` casts on plugin config, execution args, and context objects to bypass TypeScript checking. While this matches the plan's code and is pragmatic for testing plugin internals, it means the integration test provides no type-level verification that the tool's args interface matches what the opencode host would actually pass.
