### Minor
- [m1] Prototype chain is traversed for inherited keys — `src/dotpath.ts:27`

  `current = (current as Record<string, unknown>)[seg]` uses bare property access, which traverses the prototype chain. Dot-paths of `"__proto__"`, `"constructor"`, or `"toString"` resolve to their inherited values instead of returning `undefined`. For example:
  - `resolve({a:1}, "__proto__")` → `Object.prototype` (not `undefined`)
  - `resolve({a:1}, "constructor")` → `Object` (not `undefined`)

  In the `json_match` use case (read-only string comparison), this is harmless — these keys would never match an expected value, producing a safe false negative. However, it violates the implied semantics of "walking object keys" since inherited properties are not owned keys. If this function is ever reused for write operations in the future, `__proto__` traversal would become a security concern. Consider using a guard like `Object.prototype.hasOwnProperty.call(current, seg)` or documenting the behavior explicitly.

- [m2] Empty-segment edge cases not covered by tests — `test/dotpath.test.ts`

  Paths with empty segments (`""`, `"a."`, `".a"`, `"a..b"`) are not tested. The implementation handles them (returns `undefined` because `""` is not a digit-only string for arrays, and `current[""]` returns `undefined` on plain objects), but this behavior is accidental rather than designed. Adding test cases for these would document the intended behavior.

- [m3] TypeScript typecheck fails on `bun:test` — `test/dotpath.test.ts:1`

  `tsc --noEmit` fails with `TS2307: Cannot find module 'bun:test'` because `bun-types` was removed from `tsconfig.json` `types` array in Task 1. This is a pre-existing project-level issue, not introduced by Task 2, but Task 2's test file trips it. Consider adding `bun-types` back (as the plan originally specified) or suppressing the error with `@ts-ignore`.
